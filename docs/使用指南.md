# lerobot-humanoid-runtime 使用指南

本文档帮助你理解 **lerobot-humanoid-runtime** 如何运行、如何与硬件通讯、以及如何把训练好的策略部署到仿真或真机上。

> 相关文档：[README.md](../README.md) · [CALIBRATION.md](../CALIBRATION.md) · [ARCHITECTURE_CLASS_DIAGRAM.md](../ARCHITECTURE_CLASS_DIAGRAM.md)

---

## 1. 这个仓库是做什么的

`lerobot-humanoid-runtime` 是 LeRobot 双足人形平台（12 自由度、无手臂）的**执行层**：

| 能力 | 说明 |
|------|------|
| MuJoCo 仿真 | 无硬件时验证控制接口和策略 |
| 真机控制 | 通过 CAN 总线驱动 Robstride 电机，读取 IMU |
| 策略推理 | 加载 ONNX/Torch 策略，以固定频率输出关节目标 |
| LeRobot 集成 | 提供标准 `Robot` 接口，用于数据采集和生态工具 |
| 标定与安全 | 关节符号/偏置/限位、E-STOP、跳变保护 |

在 Virgileboat 工作区中的位置：

```text
lerobot-humanoid-hardware   →  BOM、装配、布线
lerobot-humanoid-model      →  URDF / MJCF 模型
lerobot-legged-zoo          →  MJLab 中训练行走策略
lerobot-humanoid-runtime    →  本仓库：仿真 + 真机部署
lerobot-humanoid-identification → 用采集数据做参数识别
```

## 2. 整体架构

本仓库有**两条并行控制路径**，共用同一套关节语义，但面向不同使用场景。

### 2.1 经典路径（策略部署推荐）

用于行走策略真机/仿真部署：

```text
┌─────────────┐     观测快照      ┌──────────┐
│  RLAgent    │ ◄──────────────── │ Bipedal  │
│ (策略推理)   │                   │ Robot    │
│  50 Hz      │ ── set_action ──► │ Controller│
└─────────────┘                   └────┬─────┘
       ▲                               │
       │ 速度指令                        │ CAN MIT 协议
┌──────┴──────┐                         ▼
│ Gamepad /   │                   ┌──────────┐
│ 固定 twist  │                   │ 12 电机   │
└─────────────┘                   │ can0+can1│
                                  └──────────┘
                                        ▲
                                  ┌─────┴────┐
                                  │   IMU    │
                                  │ I2C/UART │
                                  └──────────┘
```

**关键类：**

- `robot/bipedal_robot.py` → `BipedalRobotController`（真机）
- `robot/sim_robot.py` → `SimBipedalRobotController`（仿真，API 与真机一致）
- `control/rl_agent.py` → `RLAgent`（策略推理循环）
- `deploy/run_real_policy_sequential.py` → 分阶段真机部署脚本
- `tools/run_sim_policy.py` → 仿真中跑策略

### 2.2 LeRobot 路径（数据采集 / 生态集成）

用于与 LeRobot 框架对接（`lerobot.datasets`、遥操作、后续 imitation learning 等）：

```text
LeRobot 工具链
      │
      ▼
LeRobotHumanoid  ←→  RobstrideMotorsBus (can0/can1)
      │                    +
      └──────────────►  BNO055 IMU
```

**关键类：**

- `lerobot_humanoid_lerobot_integration/lerobot_humanoid.py` → `LeRobotHumanoid`
- `lerobot_humanoid_lerobot_integration/config_lerobot_humanoid.py` → 配置
- `tools/data_acquisition.py` → 按关节采集 LeRobot 格式数据集

## 3. 通讯机制

### 3.1 电机 CAN 总线（MIT 协议）

真机通过 **两条 CAN-FD 总线** 连接 12 个 Robstride 伺服电机：

| 总线 | 电机 ID | 对应腿 |
|------|---------|--------|
| `can0` | 1–6 | 左腿 |
| `can1` | 7–12 | 右腿 |

每条腿 6 个关节：`hipz, hipx, hipy, knee, ankle_pitch, ankle_roll`。

**协议层**（`hardware/mit_codec.py`）：

- **下行（控制）**：`pack_mit_command()` 将目标位置、速度、kp、kd、力矩编码为 8 字节 MIT 帧
- **上行（反馈）**：`decode_state_frame()` 解析位置、速度、力矩、温度

**管理命令**（`robot/root_constant.py`）：

| 命令 | 值 | 用途 |
|------|-----|------|
| `CAN_CMD_ENABLE` | 0xFC | 使能力矩 |
| `CAN_CMD_DISABLE` | 0xFD | 禁用力矩 |
| `CAN_CMD_ZERO` | 0xFE | 零点标定 |
| `CAN_CMD_CLEAR_FAULT` | 0xFB | 清除故障 |

**控制循环**（`BipedalRobotController`，默认 100 Hz）：

1. 并行读取 `can0` / `can1` 上的电机反馈
2. 将 raw motor 角度经 `MOTOR_SIGN` / `MOTOR_OFFSET_DEG` 转为模型关节角
3. 将策略/用户命令经踝关节耦合逆解转为 raw motor 目标
4. 应用限位、跳变保护、E-STOP 后批量下发 MIT 命令

启动 CAN（树莓派上）：

```bash
sudo ip link set can0 up type can bitrate 1000000 dbitrate 5000000 fd on
sudo ip link set can1 up type can bitrate 1000000 dbitrate 5000000 fd on
```

### 3.2 IMU

策略观测通常需要**投影重力**和**角速度**，来自 IMU（`imu/IMU_integration.py`）：

| 后端 | 接口 | 典型用途 |
|------|------|----------|
| `bno055` | I2C（默认地址 0x28） | 真机默认 |
| `bno085` | I2C | 备选 |
| `jy901` | 串口 `/dev/ttyAMA0` | 备选 |
| `mock` | 无硬件 | 干跑测试 |

IMU 在独立线程中以 ~100 Hz 轮询，结果通过 `get_combined_state_snapshot()` 中的 `imu` 字段提供给 `RLAgent`。

### 3.3 可视化（MeshCat）

可选的 3D 可视化（`pinocchio` + `meshcat`）：

- 真机：通过 `robot.attach_default_meshcat()` 把关节状态映射到 URDF
- 远程查看：SSH 端口转发 `ssh -L 7000:localhost:7000 user@robot-ip`，浏览器打开 `http://localhost:7000`

### 3.4 手柄指令

`apps/gamepad_controller.py` 提供 `(lin_x, lin_y, yaw_rate)` 三元组，供 `RLAgent.set_command_source()` 读取，实现行走速度指令。

## 4. 安装与环境

**要求**：Raspberry Pi 5 / Ubuntu、Python 3.13、`uv`

```bash
cd lerobot-humanoid-runtime
git submodule update --init --recursive   # 拉取 lerobot、模型子模块
uv sync                                   # 基础依赖

# 按场景安装可选组件
uv sync --extra sim       # MuJoCo 仿真
uv sync --extra policy    # Torch 策略
uv sync --extra imu       # IMU 驱动
uv sync --extra viz       # MeshCat 可视化
uv sync --extra gamepad   # 手柄
uv sync --extra tools     # 绘图/分析
uv sync --extra full      # 全部
```

## 5. 如何运行

### 5.1 仿真烟测（无策略）

验证 MuJoCo 安装和基础控制接口：

```bash
uv run python - <<'PY'
from robot.sim_robot import SimBipedalRobotController

robot = SimBipedalRobotController(control_hz=200.0, fixed_base=False)
robot.start(mode="control", auto_enable=True)
robot.start_viewer()
print("仿真已启动，关闭窗口或 Ctrl+C 停止")
PY
```

### 5.2 仿真 + 行走策略

在 MuJoCo 中验证训练好的策略：

```bash
uv run python tools/run_sim_policy.py \
  --policy-dir control/policy/codex_iteration_8 \
  --cmd-lin-x 0.20 \
  --action-scale 0.25 \
  --viewer
```

常用参数：

| 参数 | 含义 | 建议 |
|------|------|------|
| `--policy-dir` | 策略目录（含 `policy.onnx` + config） | 换成你的策略 |
| `--action-scale` | 全局动作缩放（0~1） | 从 0.1 开始逐步加大 |
| `--cmd-lin-x/y` | 前进/侧向速度 (m/s) | 先小后大 |
| `--cmd-yaw-rate` | 偏航角速度 (rad/s) | |
| `--fixed-base` | 固定基座（调试用） | 初次可开 |
| `--warmup-fixed-base-s` | 先固定基座 N 秒再释放 | 默认 1.0 |

### 5.3 真机分阶段部署（推荐）

**最安全**的真机入口，每步之间会暂停等你确认：

```bash
uv run python deploy/run_real_policy_sequential.py \
  --policy-dir control/policy/codex_iteration_6
```

四个阶段：

1. **创建机器人** → `state_only` 模式，只读状态、不下发力矩
2. **检查** → MeshCat 姿态、关节限位、IMU 方向是否与实物一致
3. **使能** → 切到 `control`，两次 `enable_all()`，发送零位姿
4. **启动策略** → 设置增益，加载 ONNX，`RLAgent.start()`

交互式调节动作幅度（终端输入）：

```text
scale 0.25    # 设为 0.25
+ / -         # 步进增减
status        # 查看当前 scale
stop          # 停止
```

### 5.4 无硬件干跑（Mock CAN）

在没有真机时验证部署脚本和策略管线：

```bash
uv run python deploy/run_real_policy_sequential.py \
  --policy-dir control/policy/codex_iteration_6 \
  --use-mock-bus \
  --no-with-imu \
  --no-with-meshcat \
  --no-with-gamepad \
  --no-pause-between-stages
```

Mock 总线由 `robstride_mock_bus.py` 模拟 12 个电机 ID 的 CAN 收发。

### 5.5 IPython 交互式调试

适合逐步手动验证：

```bash
uv run ipython
```

然后按 [`ipython_helper.py`](../ipython_helper.py) 中的片段依次执行：创建 IMU → 创建 `BipedalRobotController` → `state_only` 检查 → `control` + `enable_all` → 设增益 → 创建 `RLAgent` → `start()`。

## 6. 如何使用训练好的策略

### 6.1 策略目录结构

每个策略是一个文件夹，放在 `control/policy/<名称>/`：

```text
control/policy/my_policy/
  policy.onnx          # 必需：ONNX 推理模型
  config.yaml          # 必需：环境/观测/动作配置（以下任一名称均可）
  # 或 config.yml / model_25000_env.yaml / config.json
  gain.md              # 可选：增益调参笔记
```

运行时由 `deploy/common.py` 的 `resolve_policy_files()` 自动查找这两个文件。

### 6.2 从 lerobot-legged-zoo 导入新策略

训练仓库 [`lerobot-legged-zoo`](https://github.com/Virgileboat/lerobot-legged-zoo) 使用 MJLab 在 `Mjlab-Velocity-Flat-LeRobot-Humanoid-no-arms` 环境中训练 PPO 策略。部署到 runtime 需要两步：

**步骤 1：导出 ONNX**

在训练环境中将 checkpoint 导出为 `policy.onnx`（具体导出命令见 legged-zoo 文档）。ONNX 输入为 `[1, obs_dim]` 的 float32 向量，输出为 `[1, 12]` 的关节动作。

**步骤 2：准备 config**

训练产生的 `config.yaml`（含 `env_cfg`、观测项、关节初始位姿等）通常可直接使用。仓库内已有多个示例：

- `control/policy/codex_iteration_6/config.yaml`
- `control/policy/tao_iteration/config.yaml`

若 config 是 Isaac/MJLab 原始格式、字段结构与 runtime 不兼容，可用转换工具：

```bash
uv run python tools/convert_isaac_config_to_rlagent.py \
  /path/to/training/config.yaml \
  --output control/policy/my_policy/config.yaml
```

该工具会提取：推理频率、动作 scale、观测项（`projected_gravity`、`joint_pos`、`joint_vel`、`actions`、`command` 等）、关节初始位姿 `init_state.joint_pos`。

**步骤 3：部署验证**

```bash
# 先在仿真验证
uv run python tools/run_sim_policy.py --policy-dir control/policy/my_policy --action-scale 0.1

# 仿真 OK 后再上真机
uv run python deploy/run_real_policy_sequential.py --policy-dir control/policy/my_policy
```

### 6.3 RLAgent 如何消费策略

`RLAgent.from_files(robot, config_path, policy_path)` 的工作流程：

1. **解析 config** → 得到 `AgentSpec`：观测项列表、历史长度、推理 Hz、动作 scale、encoder bias 等
2. **加载 policy** → 支持 `.onnx`（onnxruntime）、`.pt/.pth`（Torch）、`.pkl`、自定义 loader
3. **推理循环**（默认 50 Hz）：
   - 从 `robot.get_combined_state_snapshot()` 取关节状态 + IMU
   - 按 config 中的 `policy_terms` 拼装观测向量（含历史帧）
   - `policy.infer(obs)` → 12 维动作
   - `action_scale` 缩放后转为关节目标角度
   - `robot.set_action(left=..., right=...)`

支持的观测项（`control/rl_agent.py`）：

| 观测项 | 来源 |
|--------|------|
| `projected_gravity` | IMU 四元数 |
| `base_ang_vel` | IMU 陀螺仪 |
| `base_lin_vel` | IMU 线速度（若有） |
| `joint_pos` / `joint_vel` | 关节状态（相对默认位姿） |
| `joint_torque` | 电机力矩反馈 |
| `actions` | 上一步策略输出 |
| `command` / `velocity_commands` | 手柄或固定 twist |

动作到关节角的公式：

```text
q_cmd = default_joint_pos + action × per_joint_scale × action_scale − encoder_bias
```

### 6.4 部署调参建议

| 参数 | 位置 | 建议 |
|------|------|------|
| `action_scale` | CLI 或 `agent.spec.action_scale` | **从 0 开始**，逐步加到 0.1 → 0.25 → 0.5 |
| `joint_vel_source` | `auto` / `finite_diff` / `true_value` | 真机常用 `finite_diff` |
| 电机增益 | `apply_base_gain_profile()` 或手动 `set_joint_gains()` | 参考 `gain.md` 和 `ipython_helper.py` |
| `max_command_delta_deg` | 部署脚本参数 | 限制单步关节跳变，默认 60° |

> 策略 config 里若含训练时的绝对路径（wandb 日志等），可忽略；runtime 只读取观测/动作相关字段。

## 7. 数据采集（训练数据 / 参数识别）

`tools/data_acquisition.py` 通过 **LeRobot 路径**采集逐关节激励数据，输出标准 LeRobot 数据集，供 `lerobot-humanoid-identification` 做动力学参数识别。

```bash
uv run python tools/data_acquisition.py \
  --fps 100 \
  --total-duration-s 5.0 \
  --command-mode step \
  --amplitudes-deg 0 -5 5 \
  --experiment-name experiment_5s_A
```

| 参数 | 说明 |
|------|------|
| `--command-mode step` | 阶梯信号（0° → -5° → +5°） |
| `--command-mode sinus` | 正弦激励（配合 `--sinus-amp-deg`、`--sinus-freq-hz`） |
| `--use-mock-bus` | 无硬件干跑 |
| `--datasets-root` | 输出根目录，默认 `identification/models/lerobot_humanoid/datasets` |

输出结构：

```text
identification/models/lerobot_humanoid/datasets/
  experiment_5s_A/
    acquisition_context.json   # 增益、命令配置等元数据
    manifest.yaml              # 各关节数据集路径
    <joint_name>/              # 每个关节一个 LeRobot 数据集
```

每条记录包含：关节位置/速度观测、发送的动作指令、时间戳。识别仓库读取这些数据进行仿真参数拟合。

## 8. 控制模式与安全

### 8.1 两种模式

| 模式 | 行为 |
|------|------|
| `state_only` | 只读电机反馈，不发送力矩命令。**上电后必须先用此模式** |
| `control` | 读取 + 下发 MIT 命令 |

### 8.2 安全机制

`BipedalRobotController` 内置（**不要关闭**）：

- 关节限位钳制（`JOINT_LIMITS_DEG`）
- 命令跳变保护（`max_command_delta_deg`）
- 踝关节保护（`ankle_guard`）
- E-STOP（越界/异常时停止下发）
- 阻尼模式（故障时降 kd 制动）

### 8.3 校准（部署前必做）

详见 [CALIBRATION.md](../CALIBRATION.md)。核心检查项：

1. 12 个电机在 `can0`/`can1` 上均有反馈
2. `robot/root_constant.py` 中 `MOTOR_SIGN`、`MOTOR_OFFSET_DEG` 与实物一致
3. MeshCat 中机器人姿态与实物匹配
4. 小幅度单关节运动时方向正确

## 9. 数据流时序

一次完整的策略控制周期：

```text
时间轴 ──────────────────────────────────────────────►

BipedalRobotController (100 Hz)
  │ read CAN states
  │ apply safety
  │ send MIT commands
  │         │         │         │
  ├─────────┼─────────┼─────────┤

RLAgent (50 Hz, 由 config.inference_hz 决定)
  │         get snapshot
  │         build obs vector
  │         ONNX infer
  │         set_action
  │                   │                   │
  ├───────────────────┼───────────────────┤

IMU thread (~100 Hz)
  │    │    │    │    │    │    │
  ├────┼────┼────┼────┼────┼────┤
```

`RLAgent` 和 `BipedalRobotController` 各跑独立线程，通过线程安全的 `set_action()` / `get_combined_state_snapshot()` 交互。

## 10. 常见问题

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| 内部姿态与实物不符 | 接线/符号/偏置错误 | 查 `MOTOR_SIGN`、`MOTOR_OFFSET_DEG`，见 CALIBRATION.md |
| 无电机反馈 | CAN 未启动或电机未上电 | `ip link set can0/can1 up ...`，查线束 |
| 启动即 E-STOP | 姿态越界或踝关节保护 | 把机器人摆到中立位，先 `state_only` 检查 |
| MeshCat 不更新 | 依赖未装或端口不通 | `uv sync --extra viz`，检查 7000 端口 |
| 策略加载失败 | 缺 config 或 ONNX | 确认目录下有 `policy.onnx` + `config.yaml` |
| 观测维度不匹配 | config 与 ONNX 不一致 | 用 `convert_isaac_config_to_rlagent.py` 重新生成 config |
| 机器人摔倒 | action_scale 过大 | 从 0 开始，仿真验证后再上真机 |
| LeRobot connect 超时 | 部分电机无响应 | 检查 CAN、电机 ID、电源 |

## 11. 推荐上手顺序

```text
1. uv sync --extra full
2. 仿真烟测（5.1）
3. Mock 干跑部署脚本（5.4）
4. 仿真跑策略（5.2），调 action_scale
5. 阅读 CALIBRATION.md，完成标定
6. 真机 state_only 检查 MeshCat + IMU
7. 分阶段部署（5.3），action_scale 从 0 开始
8. 需要参数识别时，跑 data_acquisition（7）
```
